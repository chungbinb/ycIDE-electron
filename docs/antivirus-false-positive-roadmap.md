# ycIDE Windows 杀软误报治理路线图

## 背景
当前用户代码编译产物为“新生成、无签名、低信誉”的 Windows 可执行文件，且由 IDE 触发“生成后立即执行”链路，容易触发启发式规则，表现为 `spawn EPERM`、产物被隔离或删除。

## 目标
- 目标 1：默认发布路径不依赖用户手工加白名单。
- 目标 2：将常见杀软误报率降到可接受水平（优先降低卡巴斯基、Defender 命中）。
- 目标 3：建立可持续反馈闭环（预检 -> 申诉 -> 规则收敛）。

## 当前基线（仓库现状）
- IDE 构建链路：Electron + Zig。
- 用户程序编译器：默认 Zig `c++`。
- 运行链路：编译后直接启动 exe。
- 发布链路：已有 `electron-builder --win`，但尚未看到明确代码签名配置。

## 分阶段改造

### Phase A（1 周）：可观测与降噪
- [ ] 在运行失败日志中增加“疑似杀软拦截”判定与引导信息。
- [ ] 统一输出启动失败上下文：`code`、`errno`、`syscall`、`path`。
- [ ] 新增用户可复制的一键诊断摘要（供误报工单提交）。

验收标准：
- 能在日志中明确区分“路径/权限问题”与“安全软件拦截疑似”。
- 用户可以复制完整诊断信息，不需要手工截图整理。

### Phase B（1-2 周）：发布产物信誉建设
- [ ] 为 IDE 主程序接入代码签名（至少 OV，推荐 EV）。
- [ ] 固定并规范 PE 元信息（CompanyName、ProductName、FileVersion、OriginalFilename）。
- [ ] 保持发布构建稳定（避免无关随机性导致签名前特征漂移）。

验收标准：
- IDE 安装包与主程序均为已签名状态。
- 多轮发布后同版本重建的哈希差异可解释。

### Phase C（2-4 周）：用户产物误报率治理
- [ ] 提供“编译后自动签名”扩展点（用户可配置证书/时间戳）。
- [ ] Debug/Release 行为分层：Release 禁止调试辅助行为落地。
- [ ] 审计运行时模板，避免高风险行为组合（可疑进程注入、自修改、异常自解压）。

验收标准：
- 启用用户签名后，主流杀软命中率显著下降。
- Release 产物行为面明显小于 Debug。

### Phase D（持续）：多引擎预检与误报申诉闭环
- [ ] 在发布前执行多引擎预检（内部样本，不上传敏感业务样本）。
- [ ] 对命中结果自动生成申诉包：样本哈希、版本、构建参数、复现步骤。
- [ ] 维护厂商工单台账（提交时间、检测名、处置状态、解除时间）。

验收标准：
- 每个发布版本都有预检记录。
- 误报可追踪并可复盘，平均处理时间持续下降。

## 仓库落地任务清单（建议）
1. `src/main/compiler.ts`
- 增加安全拦截疑似日志提示与诊断摘要。
- 为 `spawn` 失败和回退失败提供结构化文本输出。

2. `package.json`
- 增加 Windows 签名配置占位（证书路径、签名参数由环境变量注入）。

3. `scripts/`
- 新增误报申诉包生成脚本（收集 hash、版本、日志、样本路径）。
- 新增发布前安全预检脚本占位。

4. `README.md`
- 新增“杀软误报与签名建议”章节，说明发布建议与用户排查入口。

## 指标（建议）
- 误报率：每 100 次用户本地运行触发拦截次数。
- 首次误报处理时长：从出现到厂商解除。
- 已签名用户产物比例。
- 发布版本预检覆盖率。

## 风险与边界
- 无签名且频繁变化的本地编译产物，不可能做到 0 误报。
- 不应把“引导用户全局加白名单”作为默认方案。
- 需要在安全性与开发体验之间权衡（尤其是调试行为与自动执行链路）。
